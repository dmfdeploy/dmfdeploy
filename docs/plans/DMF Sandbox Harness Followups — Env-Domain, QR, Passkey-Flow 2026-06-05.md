## Task 1 — Per-env domain `<env_id>.dmf.test` for non-interactive runs

**Why:** today every test run reuses `montest.dmf.test` (from a fixed `label`),
so a stale `/etc/hosts`/resolver entry can silently point the domain at a
torn-down VM's IP. Tying the domain to the rotating env_id makes URLs unambiguous
and matches the platform's "env ids rotate" convention. **Gate to non-interactive
sandbox mode only — do NOT change interactive behavior.**

**Current state (`dmf-env/bin/init-wizard.sh`):**
- `base_domain = "${SANDBOX_LABEL}.dmf.test"` (non-interactive ~L371; interactive
  L553). `SANDBOX_LABEL` is a required input (`validate_dns_label` dies on empty,
  L291). `env_id` is auto-generated by `gen_env_id()` (`xxxx-xxxx`, L176/L526).
- dmf-init renders via `init-wizard.sh --non-interactive <answers.yaml>`; the
  answers `sandbox.label` comes from dmf-init `createnew.py` `_answers_file_contents`
  (currently hard-set; harness passes `label: "montest"` in `bootstrap.sh build_render_payload`).

**Change:**
1. `init-wizard.sh`: in **non-interactive sandbox** mode, when `sandbox.label` is
   **empty/absent**, derive `BASE_DOMAIN="${ENV_ID}.dmf.test"` (and `ENV_LABEL=$ENV_ID`)
   instead of from the label. **Ensure `env_id` is generated BEFORE base_domain is
   derived** (today `gen_env_id` at L526 may run after the label/base_domain block —
   reorder or move the derivation). Keep the non-empty-label path working
   (explicit label still wins).
2. `dmf-init/src/dmf_init/createnew.py`: allow an **empty `sandbox.label`** in the
   answers file (relax `SandboxInputs.label` min-length + `_answers_file_contents`),
   so the wizard falls back to env_id.
3. `dmf-init/test/e2e/lib/bootstrap.sh` + `profile.montest.env`: stop hard-coding
   `label: "montest"` / `SANDBOX_BASE_DOMAIN=montest.dmf.test`; pass an empty label
   so the domain becomes `<env_id>.dmf.test`. The harness already discovers the IP
   at runtime and the passkey browser uses `--host-resolver-rules` to map
   `*.<base_domain>` → that IP, so derive the base domain from the rendered env
   (read `dmf_sandbox_base_domain` from `…/envs/<env>/inventory/group_vars/all/`).

**Gotchas:** `cert_manager_cluster_domain` follows `base_domain` (fine — self-signed
local CA, regenerated per env). NetBox **site name** uses the human `ENV_LABEL`
(separate — don't conflate). DNS-label rules still apply to env_id (it's already
`xxxx-xxxx`, DNS-safe).

**Acceptance:** a non-interactive render with no label produces an env whose
`dmf_sandbox_base_domain == <env_id>.dmf.test`; interactive flow unchanged; a
harness run reaches the same green verify with the new domain.

## Task 3 — Drop the redundant username prompt from the passkey flow

**Why:** the bootstrap passkey-enrollment flow shows an **identification
(username) stage** that isn't needed — the enrollment is gated by a single-use
**invitation (`itoken`)** already bound to the user, and login uses **discoverable
(resident) passkeys** that present the user without typing a username. The prompt
is friction with no security value here.

**Current state (`dmf-infra/.../roles/stack/operator/authentik/templates/blueprints/16-passwordless-bootstrap.yaml.j2`):**
- `authentik_stages_identification.identificationstage` id
  `dmf-bootstrap-passkey-identification`, `user_fields: [username]` (~L82-90) —
  **this is the prompt to remove.**
- It's bound into the bootstrap-enrollment flow after the invitation stage
  (`authentik_stages_invitation.invitationstage` ~L73; flowstagebindings ~L115+).
- The passkey-**login** flow (validation ~L10, user-login ~L22) is separate; it
  uses discoverable WebAuthn — confirm it has no identification stage either, and
  remove if present.

**Change:** remove the `dmf-bootstrap-passkey-identification` identification stage
**and its `flowstagebinding`** from the enrollment flow so the flow goes
invitation → (webauthn register) → user-login with no username prompt. If the
flow's first stage must be an identification-class stage for Authentik to resolve
the invited user, switch it to **invitation-only / pre-filled** (use the
invitation's bound user) rather than prompting — verify the user is still resolved
from the `itoken`. Re-template + re-apply the blueprint via the authentik role.

**Test (no rollout):** re-apply the authentik blueprint/playbook against the live
env (`vertical-security/110-authentik.yml` or the authentik role play), mint a
fresh enrollment URL (`bin/get-passkey-enrollment-url.sh <env>` with the
`SOPS_AGE_KEY_FILE` from Orientation), open it, and confirm the flow no longer asks
for a username and still enrolls a passkey for `montest-op`. Then re-run
`verify-d8-hardening.yml` (it counts confirmed passkeys).

**Gotchas:** don't break the invitation→user binding (the itoken must still resolve
to `montest-op`). ADR-0028 (D8) + ADR-0010 (invitation wrapper) are the governing
decisions — keep ≥2-passkey enforcement intact; only the username *prompt* goes.
